Overview

This document provides comprehensive API documentation for the Taqql Zapier integration endpoints. The API is designed to enable seamless integration between Taqql's construction management platform and Zapier workflows.

Base URL

https://your-supabase-project.supabase.co/functions/v1/zapier
Note: Replace your-supabase-project with your actual Supabase project URL.

Authentication

The API uses token-based authentication. First authenticate to receive a token, then include it in subsequent requests.

1. Authentication Endpoint

POST /auth/

Authenticate with email/password to receive an API token.

Request Body

{
  "email": "user@example.com",
  "password": "your-password"
}

Response

{
  "person": "John Doe",
  "email": "user@example.com",
  "org": "Construction Company Inc",
  "org_id": "uuid-string",
  "key": "api-token-string",
  "expiry": "2024-02-15T10:30:00Z"
}

Error Response

{
  "error": "Invalid credentials"
}

2. Token Validation

GET /test/

Validate an existing API token.

Headers

token: your-api-token-here

Response

{
  "org": "Construction Company Inc",
  "valid": true
}

Taqql Entity Endpoints

All entity endpoints follow RESTful patterns and require authentication via the token header.

General Patterns

Method Endpoint Description
GET /taqql/{entity}/ List all entities
GET /taqql/{entity}/{id}/ Get specific entity by ID
POST /taqql/{entity}/ Create new entity
PUT /taqql/{entity}/{id}/ Update entity (full update)
PATCH /taqql/{entity}/{id}/ Update entity (partial update)

Headers (Required for all Taqql endpoints)

token: your-api-token-here
Content-Type: application/json

Webhooks

Base Path: /webhooks/

The API supports incoming webhooks for triggering Zapier workflows.

Webhook Endpoints

Method Endpoint Description
POST /webhooks/zapier/ Receive webhooks from Zapier integrations
POST /webhooks/qbo/ Receive webhooks from QuickBooks Online
POST /webhooks/time/ Receive time tracking webhooks
POST /webhooks/expense/ Receive expense-related webhooks
POST /webhooks/project/ Receive project-related webhooks

Webhook Payload Format

{
  "event_type": "timesheet_updated",
  "data": {
    "organization_id": 1,
    "entity_id": 200,
    "entity_type": "timesheet_row",
    "action": "update",
    "changes": {
      "d1_st_hours": 8.0,
      "locked": true
    }
  },
  "timestamp": "2024-01-20T15:30:00Z"
}

Error Handling

Error Response Format

{
  "error": "Error message describing what went wrong"
}

Common HTTP Status Codes

Status Code Description
200 Success
400 Bad Request (validation errors, missing fields)
401 Unauthorized (invalid or missing token)
404 Not Found (entity doesn't exist)
405 Method Not Allowed
500 Internal Server Error

Authentication Errors

  • Missing token - No token provided in headers
  • Invalid or expired token - Token is malformed or expired
  • Invalid credentials - Email/password authentication failed

Best Practices

Token Management

Store tokens securely and refresh them before expiry. API tokens are valid for 30 days from creation.

Error Handling

Always check for error responses and handle them appropriately in your integration.

Data Validation

Validate required fields before sending requests to avoid 400 errors.

Pagination

For large datasets, implement pagination on the client side to manage response sizes.

Webhooks

Use webhooks for real-time integration rather than frequent polling to improve performance.

Data Scoping

All data is automatically scoped to the authenticated user's organization. Users can only access data belonging to their organization.

Rate Limits: There are no explicit rate limits, but reasonable usage is expected. Avoid making excessive requests in short time periods.
File Attachments: Some entities (expenses, timesheet rows, POs) support file attachments. File handling is done through separate file upload endpoints and then referenced by ID in the entity creation/update requests.
